Codex Pre-Launch Fix Brief — CAD Stone Networks
Global guardrails (read these before touching anything)
Database changes: This project uses Drizzle with pnpm --filter db push-force. Never hand-write SQL migrations, never run drizzle-kit generate, never run a plain push without --force. Edit lib/db/src/schema/index.ts only, then push-force.
The runtime database is Supabase. The schema is read from SUPABASE_DATABASE_URL, not DATABASE_URL. If you need to verify data, use psql "$SUPABASE_DATABASE_URL". The local Replit Postgres replica will mislead you.
Use the helpers that already exist — do not reinvent them. Specifically:
listAccessibleJobIds, listAccessibleLeadIds, listAccessibleClientIds (server-side scope helpers)
buildScheduleListVisibilityFilter (role + personal-todo SQL filter)
assertCanViewFile / assertCanManageFile / assertCanViewScheduleItem / assertFileAccess
classifyApiError / toastApiError / apiErrorMessage from artifacts/cadstone/src/lib/api-errors.ts
writeActivity for activity_log entries
If you write a parallel implementation of any of these, you will create drift between endpoints. Refactor the helper if it doesn't quite fit; don't fork it.
Do not break existing API contracts. Same query params, same response shapes, same status codes. The OpenAPI spec at lib/api-spec/openapi.yaml is authoritative — update the spec when behavior changes.
Do not delete existing data. Several fixes touch folders/files. Migrate or categorize existing rows; don't truncate.
Run before declaring any fix done:
pnpm exec tsc --noEmit for both api-server and cadstone
pnpm --filter @workspace/api-server run test
pnpm --filter @workspace/cadstone run test
For any auth/visibility change, add a regression test to artifacts/api-server/test/ that covers admin / PM / crew / non-member access paths.
Test users: cesar@cadstone.works and anwar@cadstone.works, password Test1!. Both are admins. Don't change their roles or seed over them.
Do not introduce new dependencies (Redis, BullMQ, new ORMs, new auth libs) without flagging it first. Several findings have a tempting "just add Redis" fix that is wrong for this stage.
Critical 1 — Lead & daily-log attachments leaking through Resources
Root cause: Lead/daily-log attachment folders are created with jobId = null, which is the same shape as a "resource" folder. /resources/folders returns every jobId IS NULL folder, and assertFileAccess short-circuits as soon as the folder check passes.

Fix (do all three — defense in depth):

Add a discriminator to folders. In lib/db/src/schema/index.ts, add a scope column to folders (enum: 'resource' | 'job' | 'lead' | 'daily_log' | 'schedule_item') plus optional lead_id / daily_log_id / schedule_item_id nullable FKs. Backfill existing rows: any folder created by leads.ts → scope='lead'; by daily-logs.ts → scope='daily_log'; everything else with jobId IS NULL → scope='resource'. Backfill by querying which routes created which folders — do not guess.
Update /resources/folders (artifacts/api-server/src/routes/resources.ts:73) to filter WHERE scope = 'resource'.
Fix assertFileAccess (artifacts/api-server/src/lib/authorization.ts:473) so it does not return early on folder access alone. The order must be: if the file's parent folder is scope='lead', run the lead-access check; if scope='daily_log', run the daily-log check; only scope='resource' may rely on folder visibility alone.
Watch out:

If you only fix #2, files inside leaked folders are still reachable via direct /api/files/:id because of the early return. You must fix #3 too.
The backfill must run before the schema constraint becomes NOT NULL. Add the column nullable, backfill, then tighten.
Don't drop the jobId IS NULL allowance for resource folders — that's correct for true resources.
Critical 2 — Schedule item attachments same bypass
Root cause: Schedule attachments save into ordinary job document folders (schedule.ts:1330). The upload route correctly uses assertCanViewScheduleItem, but later read paths fall back to job/folder access and bypass schedule-item visibility.

Fix:

Use the same scope column from Critical 1. Schedule attachment folders get scope='schedule_item' and schedule_item_id = <id>.
In assertFileAccess, when scope='schedule_item', route through assertCanViewScheduleItem instead of falling back to job access.
Backfill existing schedule attachment folders by joining folders to whatever metadata currently links them to schedule items. If no link exists, add one based on existing folder naming/parent before tightening.
Watch out: Do not move existing files in object storage. Just update DB metadata.

For schedule queries: buildScheduleListVisibilityFilter(userId, role) from routes/schedule.ts:290. If it isn't already exported, extract it to artifacts/api-server/src/lib/schedule-visibility.ts and import in both places.

High 2 — Client detail leaks all jobs
Files: artifacts/api-server/src/routes/clients.ts:253, 366

Fix: In both GET /clients/:id and GET /clients/:id/jobs, intersect the returned jobs with listAccessibleJobIds(userId, role). For admins this is a no-op (returns all). For PMs/crew it limits to managed/assigned jobs.

Watch out:

Don't change client-level access (the caller already passed listAccessibleClientIds). Only filter the jobs list inside the response.
Update the OpenAPI response example if it implied "all jobs."
High 3 — Lead detail leaks all lead contacts; copy-by-sourceContactId unauthenticated
Files: artifacts/api-server/src/routes/leads.ts:474, 830

Fix:

In hydrateLead, scope availableContacts to leads in listAccessibleLeadIds(userId, role).
In the contact-copy path, before copying from sourceContactId, look up the source contact's lead_id and verify it's in listAccessibleLeadIds. If not, return 403.
Watch out: Don't break the admin path — admins must still see all available contacts.

High 4 — Workday exceptions can target unmanaged jobs
Files: artifacts/api-server/src/routes/schedule.ts:381, 2385, 2420

Fix:

appliesToAllJobs: true → restrict to role === 'admin'. Anyone else gets 403.
For an explicit jobIds: string[] payload → verify every id is in listAccessibleJobIds(userId, role) and the user has manage rights on each. Reject the whole request if any fail.
Sync logic must run for every affected job, not just the route param. If the sync helper currently takes a single jobId, loop it (in a transaction).
Watch out:

Don't remove the cross-job feature for admins — they need it for company-wide holidays.
Wrap the multi-job sync in a single transaction so partial failures don't leave inconsistent state.
High 5 — ON CONFLICT (email) doesn't match partial unique index
Files: artifacts/api-server/src/routes/auth.ts:154, lib/db/src/seed.ts:208. Index at lib/db/src/schema/index.ts:84.

Fix: Postgres requires the conflict target to match the partial index predicate exactly. Change both call sites to:

.onConflictDoUpdate({
  target: users.email,
  targetWhere: sql`${users.deletedAt} IS NULL`,
  set: { ... }
})

(or use the named index target if Drizzle's API for this version requires it).

Watch out:

Test against Supabase, not the local replica. Re-running seed and re-registering the same email should hit the upsert path cleanly with no there is no unique or exclusion constraint matching the ON CONFLICT specification error.
Do not drop the partial index — it's correct (we want soft-deleted users to free their email).
Medium 1 — Autoscale + in-memory rate limits / JTI replay
Files: .replit:9, artifacts/api-server/src/lib/rate-limit.ts:17, artifacts/api-server/src/lib/auth.ts:188

Fix (preferred — simplest, no new infra): Change .replit deployment target from autoscale to vm (single Reserved VM). With one process, the in-memory stores are correct. Document this constraint in replit.md.

Alternative (only if VM isn't acceptable): Move JTI replay protection to Postgres with a signed_url_jti_used table (jti TEXT PRIMARY KEY, used_at TIMESTAMPTZ, with a periodic cleanup). Move rate limits to a rate_limit_counters table with a sliding window. Do not introduce Redis.

Watch out:

If staying on autoscale without backing the stores: every multi-instance fix needs both rate limiting and JTI persisted, or signed URLs become replayable across instances.
Do not silently change the deploy target — flag it for the user.

Medium 3 — multer.memoryStorage 200MB DoS risk
Files: artifacts/api-server/src/routes/files.ts:33, routes/leads.ts:48, routes/daily-logs.ts:50

Fix: Switch to multer.diskStorage() with a temp directory, then stream the file to object storage and delete the temp file. Lower per-file limit to something realistic (e.g. 100 MB) and per-request total. Add a try/finally that always unlinks the temp file.

Watch out:

Don't just reduce the size limit — a coordinated 20-file × 100 MB upload still exhausts memory.
Do not change the API contract (multipart field names stay the same).
The temp directory must be writable by the process and excluded from git (see Lower 2).
