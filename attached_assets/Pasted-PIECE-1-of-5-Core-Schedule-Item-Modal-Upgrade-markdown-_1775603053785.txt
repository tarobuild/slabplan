### 5. Predecessors & Links Full Integration

Ensure the Predecessors sub-tab (from Piece 1) actually affects scheduling:

- **Finish-to-Start (FS)**: The successor item cannot start until the predecessor finishes. If predecessor End Date changes, successor Start Date auto-adjusts
- **Start-to-Start (SS)**: The successor item starts when the predecessor starts. If predecessor Start Date changes, successor Start Date follows
- **Lag**: Adds X days delay between the predecessor and successor. E.g., FS with Lag 2 means successor starts 2 days AFTER predecessor ends
- **Gantt view**: Draw dependency arrows/lines between linked items showing the relationship
- **"Track Conflicts" action** (from More Actions menu): Scan all items and highlight any scheduling conflicts where predecessor logic is violated

- Workday exceptions should be stored at the job level and optionally at the company level ("All jobs")
- Predecessor relationships need a join table: `schedule_item_predecessors` with columns: item_id, predecessor_id, type (FS/SS), lag_days

## API Notes
- Daily Log Settings: `GET/PUT /api/daily-logs/settings`
- Custom Fields: `POST/GET/PUT/DELETE /api/daily-logs/custom-fields`
- Schedule Templates: `GET /api/schedule-templates`, `POST /api/jobs/:jobId/schedule/import-template`
- Schedule History: Implement as an audit log — every mutation to schedule items writes a history record with: action type, user_id, timestamp, old_values (JSON), new_values (JSON)
- PDF Export: Consider a server endpoint `GET /api/jobs/:jobId/schedule/export-pdf` that renders the schedule server-side, or use client-side html2pdf.js
